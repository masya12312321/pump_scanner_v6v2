| 1 | SQLite открывал новое соединение на каждый вызов → file-lock contention при 10 воркерах | `db_pool.py` — singleton-соединение, WAL mode, serialised writer lock |

## Новый функционал (по запросу)

- **`outcome_tracker.py`** — каждые 5 мин проверяет алерты старше 30 мин, тянет текущий MCap, классифицирует x2/x5/x10/rug/dead, вызывает `record_outcome()` — замыкает цикл ML.
- **Multi-dex агрегация** — `get_dex_data()` считает `pair_count`, флаги `on_raydium`/`on_meteora`. Переход на Raydium = `+12` к confidence (зелёный флаг).
- **Команда `/last`** — 5 последних алертов. Если `matplotlib` установлен, присылает горизонтальный bar-chart с PnL% и цветовой кодировкой (🚀 зелёный x2+, 🔴 красный — слив). Если matplotlib нет — текстовый fallback с emoji, бот не падает.
- **Dead-token hard penalty** — `volume_5m == 0 and age_min > 3` → `-35` confidence, риск-флаг.
- **Cache используется реально** — `get_mint_info`, `check_rugcheck`, `get_dex_data`, `get_largest_accounts` все проверяют `CacheManager` перед HTTP-запросом.
- **retry_manager bug fixed** — `retry_scheduler` теперь получает `new_token_queue`, не `alert_queue`.

## Команды бота

`/start /stats /queue /filters /blacklist /topcreators /retry /health /cache /weights /last`

**Автоторговля**: `/trading /autotrade /paper /amount /tp /sl /maxpos /minconf /dailylimit /positions /pnl /wallet`

## Архитектура

```
pump.fun WS → new_token_queue → 10 analysis workers (cached API calls)
                                      ↓
                              alert_queue → 2 alert workers → Telegram
                                      ↓                ↓
                                      ↓        trading_engine.maybe_open_position()
                                      ↓                ↓ (если autotrade=on и confidence ≥ порога)
                                      ↓        открыта позиция (paper или real)
                          exit_engine.register() → live monitoring (30 мин)

outcome_tracker (5 мин)  → проверяет старые алерты → record_outcome → creator_scoring
self_learning (24ч)      → пересчитывает веса → calculate_scores (в реалтайме)
retry_scheduler (10с)    → new_token_queue (не alert_queue — исправлено)
cache_cleanup (60с)      → чистит expired записи
trading_engine.monitor_loop (15с) → проверяет TP/SL по всем открытым позициям
```

## Автоторговля (`trading_engine.py` + `wallet_manager.py`)

Бот может сам открывать и закрывать позиции по своим же сигналам — без участия человека на каждую сделку.

- **Paper-режим (по умолчанию)** — полная симуляция. Сделки пишутся в БД (таблица `positions`) с реальными ценами с Dexscreener, но реальные деньги не тратятся. Позволяет оценить стратегию перед тем как доверить ей реальный кошелёк.
- **Real-режим (`/paper off`, требует подтверждения)** — реальные сделки на Solana через **PumpPortal Local Transaction API**: PumpPortal только собирает неподписанную транзакцию buy/sell по бондинг-кривой pump.fun, подписываем её локально ключом из `WALLET_PRIVATE_KEY` и отправляем через собственный Helius RPC. Приватный ключ не передаётся ни PumpPortal, ни кому-либо ещё.
- **Вход**: фиксированная сумма SOL на сделку (`/amount`), срабатывает только если автоторговля включена (`/autotrade on`) и `confidence` сигнала ≥ отдельного, более строгого порога (`/minconf`, по умолчанию 55 — выше базового порога алерта в 35).
- **Выход**: фиксированные Take Profit / Stop Loss в % (`/tp`, `/sl`) от цены входа. Проверяются каждые 15 секунд (`trading_engine.monitor_loop`) через тот же кэшированный `get_dex_data()`, что использует движок анализа.
- **Защита от потерь**: `/maxpos` — лимит одновременно открытых позиций; `/dailylimit` — суточный kill-switch (если накопленный убыток за 24ч превышает лимит, новые позиции не открываются до следующего дня).
- **Переключение на реальные деньги защищено двойным подтверждением**: `/paper off` → предупреждение → `/paper off confirm` в течение 2 минут, и только если `WALLET_PRIVATE_KEY` задан.

⚠️ **Это не финансовый совет и не гарантия прибыли.** Мемкоины pump.fun — крайне высокорисковый актив: подавляющее большинство токенов теряет почти всю стоимость в первые часы/дни. TP/SL ограничивают размер убытка по одной сделке, но не убирают риск полной потери депозита, отведённого под автоторговлю. Рекомендуется: начать с paper-режима, прогнать стратегию хотя бы несколько дней/недель, и в real-режиме использовать только сумму, потерю которой вы готовы принять.

## Известные ограничения (честно)

- `pump.fun` WebSocket — публичный эндпоинт pumpportal.fun, может менять схему сообщений без предупреждения.
- Helius free tier — 100k запросов/день, при высокой нагрузке может потребоваться платный план.
- Bundle detection — эвристика на % держания, не отслеживает on-chain связи кошельков (для этого нужен граф-анализ транзакций, отдельный модуль).
- Self-learning — простой gradient step на BCE, не полноценная логрегрессия с регуляризацией. Подходит для медленной адаптации весов, не для точного предсказания.
- **Автоторговля**: цена входа в real-режиме оценивается через Dexscreener с задержкой ~3 сек после отправки транзакции (для учёта slippage и фактического исполнения по бондинг-кривой), а не парсится из самой транзакции — при резком движении цены в эти 3 сек оценка PnL может быть немного неточной. PumpPortal Local API рассчитан на токены, ещё торгующиеся на бондинг-кривой pump.fun; после миграции на Raydium механика свопа может отличаться.

## Редизайн алертов (по фидбеку)

- **Бренд**: заголовок теперь `🧠 DEXMIND SIGNAL` вместо родового `PUMP SIGNAL`.
- **Иерархия**: Безопасность → Рынок → Риски/Плюсы → Прогноз и стратегия. Сначала "можно ли это покупать", потом "что происходит сейчас", потом "где может облажаться", в конце — потенциал и тайминг выхода.
- **Дедуп**: ликвидность, топ-1%, объём/MCap, возраст, RugCheck-балл и Creator Score больше не повторяются дважды (один раз явным числом в соответствующем блоке, не плюс копия в списке флагов).
- **x50/x100 убраны из текста**: остаются в логике расчёта `peak_mcap_est`, но не показываются — доли процента не несут пользы, только шум. В сообщении только x2/x5/x10.
- **ATH и Объём(15м) скрываются для токенов младше 3 минут** — у них физически не может быть ни исторического максимума, ни 15-минутного окна объёма, так что поля не засоряют экран.

## Live X-Multiplier Tracking (по референсу скриншота)

Новый модуль `x_milestone_tracker.py` — после отправки сигнала бот не теряет токен из виду. Каждые 25 секунд опрашивает MCap живых алертов (до 2 часов после сигнала) и шлёт отдельное короткое сообщение каждый раз, когда цена пересекает новый множитель:

```
🧠 DEXMIND
🪙 LEVELZE
⚡ x2.0 за 25m
🕐 22 06:38
```

- Пороги: x2 / x3 / x5 / x10 / x20 / x50 / x100 — без повторов одного и того же порога.
- `alerts.max_x_hit` в БД хранит максимально достигнутый множитель — следующее сообщение шлётся только если фактический X превысил уже отправленный.
- Формат времени совпадает с референсом: `25m` для минут, `6h:11m` для часов.
- Работает параллельно с `outcome_tracker.py` (тот пишет финальный исход x2/x5/x10/rug/dead в БД для ML раз в 5 мин на токенах старше 30 мин) — milestone tracker даёт мгновенную обратную связь в чат, outcome tracker замыкает цикл обучения.

## КРИТИЧЕСКИЙ ФИКС: ложное срабатывание на USDC

Бот ошибочно проанализировал USDC (`EPjFWdd5AufqSSqeM2qN1xzybapC8G4wEGGkZwyTDt1v`) как свежий мемкоин и выдал прогноз роста с $74.8B до $102B. Причина — никакого верхнего предела по MCap не было, любой токен с pump.fun WebSocket уходил в полный анализ независимо от размера.

**Два пояса защиты, оба в `config.py` + `analysis.py` + `main.py`:**

1. **`KNOWN_MAJOR_TOKENS`** — хардкод-сет contract address'ов известных стейблкоинов и мейджоров (USDC, USDT, wSOL, mSOL, wETH, wBTC, JUP, BONK). Проверка происходит первой, без единого сетевого запроса — `ca in config.KNOWN_MAJOR_TOKENS` отбрасывает токен за микросекунды.
2. **`MAX_SAFE_MCAP_USD = $2,000,000`** — жёсткий потолок. Если MCap токена превышает $2M, бот его не анализирует как мемкоин. Проверка стоит дважды: в `analysis.py` сразу после первого запроса к Dexscreener (до того как потрачены Helius/RugCheck лимиты на токен, который заведомо отсеется) и повторно в `main.py` перед отправкой алерта (защита от ситуации, когда MCap вырос за секунды между анализом и отправкой).

## Компактный формат сообщений (по фидбеку)

- **Убран дубль Rug Score / RugCheck** — это было одно и то же число под двумя разными подписями. Остался только `Rug Score` (внутренний расчёт бота, уже включает данные RugCheck внутри формулы).
- **Числа округляются**: `fmt_usd()` — `$74,884,784,553` → `$74.9B`, `$74,800` → `$74.8k`. Применяется к MCap, ликвидности, объёму, ожидаемому пику.
- **Нулевой momentum схлопывается**: если все интервалы (1м/3м/5м) меньше ±1% — строка не показывает три нуля, просто `⏳ Накапливаем`.
- **Блоки "Плюсы" и "Риски" убраны целиком.** Каждый фактор показан один раз явным числом в нужном разделе (Безопасность/Рынок) — не дублируется отдельным списком.

## ФИКС: forecast_hit никогда не срабатывал

Баг: `alert_worker` в `telegram_bot.py` вызывал `save_alert()` без аргумента `predicted_peak_mcap` — в БД всегда писался `0.0`. Из-за этого `x_milestone_tracker.py` не мог сравнить текущий MCap с прогнозом (`current_mcap >= predicted_peak`, где `predicted_peak` всегда 0) — уведомление о достижении спрогнозированного пика никогда не приходило, особенно критично для слабых сигналов (Grade C/D), где прогноз ниже x2 и круглые пороги (x2/x3/x5...) не подходят как триггер.

Фикс: добавлена функция `get_predicted_peak_mcap(sig)` в `telegram_bot.py`, которая считает `peak_mcap_est` через `calculate_x()` независимо от рендера текста сообщения. `alert_worker` теперь передаёт это значение в `save_alert(..., predicted_peak_mcap=predicted_peak)`. Проверено на точных цифрах из реального сигнала ($58.6k → прогноз $80.0k, совпадает с тем что бот показал в тексте).
